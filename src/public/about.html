Khai

Operational boundaries, safety checks, and data-handling reminders

What Khai Can Do

  • Log into configured sites

    Khai can use stored credentials from config/credentials.json to access authenticated pages for crawling, actions, watches, and some audits.

  • Test multiple viewports

    The crawler supports desktop, tablet, and mobile viewport presets. Some REST-only features such as Lighthouse use their own desktop/mobile handling.

  • Persist reports and screenshots locally

    JSON reports are written under reports/ and screenshots under screenshots/ so runs can be reviewed after completion.

  • Send webhook notifications

    Several async jobs can POST their final results to a webhook target, and watches can notify when changes are detected.

Current Limits

  • No autonomous purchases

    Khai can fill payment forms and stage purchase flows, but checkout completion still requires explicit confirmation through the purchase confirmation path.

  • No source-code editing

    The Khai server tests sites and records outputs. It does not patch your application code or deploy changes.

  • Local service only

    The Express server binds to 127.0.0.1. It is intentionally local unless you change the code or run it behind another service yourself.

  • CAPTCHA and anti-bot flows may block tests

    Khai does not include a CAPTCHA-solving system. Login or checkout steps guarded by anti-bot controls may still require manual intervention.

  • Some features are REST-only

    Not every Express route is exposed through MCP. Suites, communications, scheduler controls, and several advanced jobs currently require the REST API.

Security Controls

  • Optional API-key protection

    If KHAI_API_KEY is set, requests to /api/* must include an X-Khai-Key header. The health endpoint and static pages remain open.

  • Restricted dangerous actions

    The evaluate action is disabled unless KHAI_ALLOW_EVAL=true. Cross-host navigation is blocked unless KHAI_ALLOW_EXTERNAL_NAV=true.

  • Webhook target validation

    Webhook URLs must use http or https. Targets resolving to private or reserved IP ranges are rejected. Deliveries retry up to three times with backoff.

  • Webhook signing

    If KHAI_WEBHOOK_SECRET is set, outbound webhook bodies are signed with HMAC-SHA256 in the X-Khai-Signature header.

Data Handling Reminders

  • Credentials and test data stay local by default

    Credentials are loaded from a local config file. Reports and screenshots may still contain sensitive data, so treat the workspace as sensitive storage.

  • Screenshots can capture regulated data

    If you test healthcare, financial, or customer systems, screenshots and JSON reports may include PHI, PII, or payment-adjacent details. Store and clean them accordingly.

  • Use test accounts and test payment data where possible

    This reduces compliance exposure and lowers the risk of unintended side effects during automated runs.

Authorization and Compliance

  • Only test systems you are allowed to test

    Use Khai only against systems you own or have explicit authorization to assess.

  • Review domain-specific obligations

    Healthcare, finance, and customer-data workflows may trigger HIPAA, PCI-DSS, GDPR, CCPA, or contractual obligations. Khai does not remove those requirements.

  • Maintain retention discipline

    Old screenshots, HAR files, crawl reports, watch histories, and suite outputs should be reviewed and removed according to your retention policy.
