Overview

Khai is a local AI lifecycle agent for automated website testing, security auditing, credential management, and communication monitoring. It runs on your machine at localhost:3001 and uses Puppeteer for browser automation.

🕷️

Crawl & Test

Automated deep crawling with login support, issue detection, and full-page screenshots at any viewport.

🛡️

Security Audit

14-category security and functional audit: headers, SSL, CORS, auth bypass, sensitive paths, rate limits, and more.

🎯

Custom Actions

Execute precise action sequences: navigate, click, fill forms, extract text, run JS, and take screenshots.

📸

Visual Regression

Pixel-level screenshot comparison with diff images. Detect unexpected visual changes between deployments.

🔄

Flow Testing

Multi-step user journey testing with assertions. Test login flows, checkout, registration, and custom workflows.

💥

API Fuzzing

Fuzz API endpoints with XSS, SQL injection, NoSQL, type confusion, and boundary value payloads. 135+ tests per endpoint.

📝

Form Fuzzing

⚡

🔗

Link Checker

Crawl all pages, discover every link (a, img, script, link), check for broken URLs with concurrent HEAD/GET requests.

Website Crawling & Testing

The core test engine logs into a site with stored credentials and crawls every discoverable link up to a configurable depth. For each page it checks:

  • HTTP Status — flags 4xx and 5xx errors
  • Load Time — warns on pages over 5 seconds
  • Broken Images — scrolls page to trigger lazy-loaded images, checks natural dimensions
  • JavaScript Errors — captures console errors and uncaught exceptions
  • Failed Requests — logs network failures
  • Accessibility — missing alt text, empty buttons/links, unlabeled inputs, heading hierarchy
  • Form Security — CSRF token presence on POST forms
  • Screenshots — full-page capture at configured viewport

Viewports: desktop (1920x1080), tablet (768x1024), mobile (375x812)

Site Audit Engine NEW

A comprehensive 14-category audit that tests functional correctness, security posture, SEO, and performance. Runs with or without Khai — unauthenticated tests use direct HTTP requests (no browser needed), while authenticated tests leverage Khai's login capabilities.

1

Public Pages No Khai

Verify every public page returns expected status codes and contains expected content. Flag missing pages or unexpected errors.

2

Redirects No Khai

Follow redirect chains for old URLs and verify they land at the correct destination with proper status codes (301/302).

3

Security Headers No Khai

Check for required headers (X-Frame-Options, X-Content-Type-Options, HSTS) and recommended ones (CSP, Referrer-Policy, Permissions-Policy). Evaluates CSP quality and flags unsafe-inline/unsafe-eval.

4

Cookie Security No Khai

Inspect Set-Cookie headers for Secure, HttpOnly, and SameSite flags. Warns on session cookies missing protections.

5

CORS Policy No Khai

Send requests with a malicious Origin header. Detect wildcard CORS, origin reflection, or dangerous credentials + wildcard combinations.

6

Auth Bypass No Khai

Attempt to access protected pages and API routes without authentication. Verify proper redirects to login or 401/403 responses.

7

Sensitive Paths No Khai

Probe for exposed config files (.env, .git, package.json, phpinfo, etc.). Flags any that return 200 with sensitive content.

8

API Endpoints No Khai

Health check known APIs. Test that error responses don't leak stack traces or database information. Verify unknown routes return clean 404s.

9

Rate Limiting No Khai

Send 20 rapid-fire requests to detect rate limiting (429 responses). Also tests login endpoint for brute-force protection.

10

SSL/TLS No Khai

Verify HTTPS, check HTTP-to-HTTPS redirect, inspect certificate validity and expiration, and confirm issuer details.

11

SEO No Khai

Check title tags, meta descriptions, Open Graph tags, canonical links, and H1 presence on key pages.

12

Performance No Khai

Measure response times, body sizes, compression, and cache headers for public pages.

13

Authenticated Pages Needs Khai

Log in as admin and patient roles, crawl authenticated areas, verify expected pages load correctly, and capture screenshots.

14

Authorization Needs Khai

Log in as one role (e.g. patient) and attempt to access another role's pages (e.g. admin). Verify access is denied.

Security Testing Details

The audit engine runs these security checks automatically. Here's what each category looks for:

🔓

Auth Bypass Checks

Direct URL access to /admin, /dashboard, protected API routes. Verifies redirect-to-login or 401/403. Checks tRPC procedures require auth.

📋

Info Leak Detection

Scans error responses for stack traces, database names (Prisma/Postgres), and internal paths. Checks X-Powered-By and Server version headers.

🌐

CORS Misconfiguration

Tests with evil origin to detect reflection attacks. Flags wildcard + credentials combos. Checks both root and API endpoints.

🍪

Cookie Hardening

Session cookies: Secure flag on HTTPS, HttpOnly to prevent XSS theft, SameSite to prevent CSRF. Checks all cookies, not just session.

🔒

SSL/TLS Posture

Certificate expiry warnings (30-day threshold), HTTP→HTTPS enforcement, HSTS quality (includeSubDomains, max-age ≥ 1 year).

🚦

Rate Limit Testing

20 rapid requests to API endpoints, 10 bad login attempts to auth endpoint. Looks for 429 status or request failures indicating throttling.

📁

Sensitive File Exposure

Probes 20+ paths: .env, .git, package.json, phpinfo, wp-admin, server-status, .htaccess, tsconfig, DS_Store, and more.

🛡️

CSP Evaluation

Checks for default-src, warns on unsafe-inline without nonce, flags unsafe-eval. Reports missing CSP as recommendation.

Visual Regression Testing NEW

Compare screenshots pixel-by-pixel to catch unexpected visual changes between deployments. Uses pixelmatch for accurate diff detection with configurable tolerance.

How It Works

1. Set a baseline from a known-good test run.
2. After changes, take new screenshots.
3. Compare baseline vs current — get a diff image with changed pixels highlighted in red.

Capabilities

Handles different-sized images (pads smaller to match). Reports: matched, changed, missing, and added pages. Sorts diffs by severity (largest change first). Threshold from 0 (strict) to 1 (loose).

Example

# Set baseline from a Khai test run
curl -X POST http://localhost:3001/api/advanced/visual/set-baseline \
  -H "Content-Type: application/json" \
  -d '{"testId": "abc123"}'

# Compare current screenshots against baseline
curl -X POST http://localhost:3001/api/advanced/visual/compare \
  -H "Content-Type: application/json" \
  -d '{"threshold": 0.1}'

Flow Testing NEW

Define multi-step user journeys as JSON configs and run them with assertions at each step. Perfect for testing login flows, checkout processes, form submissions, and navigation paths.

Action Types

navigate, click, fill, select, wait, screenshot, assertText, assertUrl, assertVisible, assertNotVisible

Configurable

Flows defined in config/flows/{site}.json. Run one flow or all flows. Login support for authenticated journeys. Full step-by-step results with screenshots.

Flow Config Example

{
  "site": "yoursite.com",
  "flows": [{
    "name": "eligibility-form",
    "steps": [
      { "action": "navigate", "url": "/eligibility" },
      { "action": "assertText", "selector": "h1", "expected": "eligibility" },
      { "action": "fill", "selector": "#name", "value": "Test User" },
      { "action": "click", "selector": "button[type=submit]" },
      { "action": "screenshot", "name": "after-submit" }
    ]
  }]
}

API

# Run all flows for a site
curl -X POST http://localhost:3001/api/advanced/flows/run \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com"}'

# Run a specific flow
curl -X POST http://localhost:3001/api/advanced/flows/run \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com", "flowName": "eligibility-form"}'

API Fuzzing NEW

Fuzz API endpoints with 135+ payload variants per endpoint. Generates XSS, SQL injection, NoSQL injection, type confusion, boundary values, special characters, missing fields, and prototype pollution payloads.

Payload Categories

20 XSS payloads, 10 SQL injection, 5 NoSQL injection, 7 special chars, 10 type confusion per field, 8 boundary values for numbers, plus missing fields and prototype pollution.

Detection

Detects: server crashes (500), stack trace leaks, SQL error messages, XSS reflection in HTML responses, NoSQL operator injection acceptance. Severity: critical or high.

Example

curl -X POST http://localhost:3001/api/advanced/fuzz/api \
  -H "Content-Type: application/json" \
  -d '{
    "baseUrl": "https://yoursite.com",
    "endpoints": [{
      "path": "/api/eligibility",
      "method": "POST",
      "body": {"email": "test@test.com", "state": "FL", "age": 25}
    }]
  }'

Form Fuzzing Needs Browser

Automatically discovers every form on a page, identifies field types (email, number, tel, password, text, textarea, select), and submits type-appropriate adversarial inputs. Reloads the page between each attempt for clean state.

What It Tests

Empty values, oversized inputs (5000+ chars), XSS payloads, SQL injection, special characters, null bytes, emoji, template injection. Field-type-aware: email fields get email-specific malformed inputs, numbers get NaN/Infinity, etc.

Outcomes

crash — page went blank or showed error boundary/500.
xss_reflected — payload found unescaped in DOM.
rejected — validation caught the bad input.
accepted — input went through without validation.

Example

curl -X POST http://localhost:3001/api/advanced/fuzz/forms \
  -H "Content-Type: application/json" \
  -d '{
    "baseUrl": "https://yoursite.com",
    "pages": ["/eligibility", "/register", "/login"]
  }'

Lighthouse Performance NEW

Measures Core Web Vitals using Chrome DevTools Protocol — no external dependencies needed. Scores each metric as good/needs-improvement/poor using Google's thresholds.

Metrics Measured

LCP (Largest Contentful Paint): good <2.5s, poor >4s
CLS (Cumulative Layout Shift): good <0.1, poor >0.25
TTFB (Time to First Byte): good <800ms, poor >1800ms
FCP (First Contentful Paint): good <1.8s, poor >3s
Load Time: good <3s, poor >5s

Features

Mobile and desktop viewport presets. Audit single page or batch. Per-page detailed metrics plus overall summary. Resource timing breakdown (scripts, stylesheets, images, fonts, API calls).

Example

curl -X POST http://localhost:3001/api/advanced/lighthouse \
  -H "Content-Type: application/json" \
  -d '{
    "viewport": "mobile",
    "pages": [
      {"url": "https://yoursite.com", "name": "Home"},
      {"url": "https://yoursite.com/blog", "name": "Blog"},
      {"url": "https://yoursite.com/login", "name": "Login"}
    ]
  }'

Link Checker NEW

Crawls your site with Puppeteer to discover all links (anchors, images, scripts, stylesheets), then validates each unique URL with HEAD/GET requests. Concurrent checking with configurable parallelism.

What It Finds

Broken links — 404s, 500s, DNS failures, timeouts, SSL errors, connection refused.
Slow links — responses over 3 seconds.
Redirect chains — tracks full redirect path for each URL.

Report

Separates internal vs external links. Broken links sorted by impact (found on most pages first). Shows which page each broken link was discovered on. Configurable: max pages to crawl, concurrency, timeout.

Example

curl -X POST http://localhost:3001/api/advanced/links/check \
  -H "Content-Type: application/json" \
  -d '{"baseUrl": "https://yoursite.com", "maxPages": 30, "concurrency": 10}'

Audit Scheduler NEW

Run audits on a recurring schedule. Automatically detects regressions by comparing each run with the previous one. Sends webhook and email notifications when things go wrong.

Regression Detection

Compares each run to the last: more failures, fewer passes, more warnings, or pass-rate drops >5% are flagged as regressions. Notification subjects prefixed with [REGRESSION] when detected.

Persistence

Schedules and history saved to config/schedules.json. Survives restarts. Retains last 50 runs per schedule. Start/stop individual schedules or all at once.

Example

# Create a schedule (every 6 hours)
curl -X POST http://localhost:3001/api/advanced/scheduler \
  -H "Content-Type: application/json" \
  -d '{
    "site": "yoursite.com",
    "interval": 21600000,
    "useKhai": true,
    "webhookUrl": "https://hooks.slack.com/services/YOUR/WEBHOOK/URL"
  }'

# Start a schedule
curl -X POST http://localhost:3001/api/advanced/scheduler/{id}/start

# View run history
curl http://localhost:3001/api/advanced/scheduler/{id}/history?limit=5

Communications Monitor

Monitor incoming email, SMS, and fax messages during testing. Useful for:

  • Capturing verification codes for 2FA login flows
  • Verifying email notifications are sent correctly
  • Checking SMS delivery for appointment reminders
  • Monitoring fax transmissions

Configure API endpoints in config/credentials.json under the communications section. The monitor polls at configurable intervals and auto-extracts numeric codes from messages.

Purchase Testing

When Khai encounters a checkout page during crawling, it:

  1. Detects payment form fields automatically
  2. Fills card details from stored test cards
  3. Pauses and asks for your explicit confirmation
  4. Only completes the purchase if you approve

Configure test cards in config/credentials.json under the payment section. Always use test/sandbox card numbers (e.g. 4111111111111111) during testing.

Password Rotation

Automate password changes with full audit trail. Khai navigates to the change-password page, fills old and new passwords, and verifies the update. Supports custom selectors for any site's password change form.

Quick Start Examples

Run a full audit (no Khai needed)

# Start an audit of yoursite.com using its profile
curl -X POST http://localhost:3001/api/audit/start \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com"}'

# Check audit status
curl http://localhost:3001/api/audit/{auditId}/status

# Get full results
curl http://localhost:3001/api/audit/{auditId}/results

Run audit with Khai (authenticated tests)

# Include authenticated + authorization tests
curl -X POST http://localhost:3001/api/audit/start \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com", "useKhai": true}'

Run only security tests

# Run specific categories only
curl -X POST http://localhost:3001/api/audit/start \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com", "categories": ["securityHeaders", "cookieSecurity", "cors", "authBypass", "sensitivePaths", "ssl", "rateLimiting"]}'

Audit a new site (no profile)

# Provide baseUrl directly — runs with default checks
curl -X POST http://localhost:3001/api/audit/start \
  -H "Content-Type: application/json" \
  -d '{"baseUrl": "https://example.com"}'

Quick crawl test

# Crawl as admin with depth 2
curl -X POST http://localhost:3001/api/test/start \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com", "account": "admin", "maxDepth": 2}'

Take a screenshot of a specific page

# Navigate to admin page and capture screenshot
curl -X POST http://localhost:3001/api/actions/execute \
  -H "Content-Type: application/json" \
  -d '{"site": "yoursite.com", "account": "admin", "actions": [
    {"type": "navigate", "target": "https://yoursite.com/admin"},
    {"type": "wait", "selector": "body", "timeout": 10000},
    {"type": "screenshot", "name": "admin-dashboard"}
  ]}'

_template.json

Copy this to create a profile for any new site. All fields documented with examples.

Profile structure

{
  "siteName": "example.com",
  "baseUrl": "https://example.com",
  "publicPages": [{ "path": "/", "expectedStatus": 200, "expectedContent": ["Welcome"] }],
  "redirects": [{ "from": "/old", "to": "/new" }],
  "protectedPages": [{ "path": "/admin", "redirectsTo": "/login" }],
  "protectedAPIs": [{ "path": "/api/user", "expectedStatus": 401 }],
  "apiEndpoints": [{ "path": "/api/health", "expectedStatus": 200 }],
  "sensitivePaths": ["/.env", "/.git/config"],
  "securityHeaders": { "required": [...], "recommended": [...] },
  "authenticatedTests": { "admin": { "account": "admin", "pages": ["/admin"] } },
  "authorizationTests": [{ "loginAs": "patient", "accessPath": "/admin", "expectDenied": true }]
}

To add a new site: copy _template.json, rename to yoursite.com.json, and customize the paths and expected values.
